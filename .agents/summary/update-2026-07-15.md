## 0. Bugs

### 0.1 · El segundo i18n está roto: las imágenes ignoran el idioma

- **Qué**: `internals/i18n.ts` resuelve el idioma leyendo un `let currentLang`
  de módulo que **nadie inicializa** — `setLanguage()` no se llama en ninguna
  parte del árbol. Siempre vale `'en'`. Lo usa sólo `Image.astro`, así que una
  imagen con `src` o `alt` traducible (`{ en, es }`) sale **siempre en inglés**,
  aun en las páginas `/es/`.
- **Dónde**: `src/internals/i18n.ts` · `src/components/atoms/Image.astro:4,17,18`
- **Por qué**: es un bug silencioso (nada falla, sólo traduce mal) y, de fondo,
  el `let` de módulo es justo lo que el AGENTS.md del mono prohíbe. `Image` ya
  vive en páginas que conocen el idioma por la URL; sólo tiene que usar el
  sistema que ya funciona.
- **Esfuerzo**: S · **Riesgo**: bajo

```astro
---
// Image.astro — antes: t() lee un idioma global que nadie fija
import { t } from '../../internals/i18n';
const { src, alt } = ...(t(value.src!), t(value))...;

// después: el mismo useTranslations que usa todo lo demás
import { useTranslations } from '@/i18n';
const { tr } = useTranslations(Astro.url);
const { src, alt } = ...(tr(value.src!), tr(value))...;
---
```

Hecho esto, `internals/i18n.ts` queda sin usuarios y se borra entero (§1.1).

## 2. Simplicidad

### 2.1 · Un helper para localizar el body

- **Qué**: las mismas dos líneas se repiten en las 6 páginas `[slug]`:

  ```ts
  const content = Array.isArray(x.body) ? x.body : [x.body];
  const localizedContent = content[languages.indexOf(lang)] || content[0] || '';
  ```
- **Dónde**: `pages/[lang]/{blog,career,projects,experiments,talks}/[slug].astro`
- **Por qué**: seis copias de una regla ("el body es un bloque por idioma,
  separado por `---`") que debería tener un nombre. Un `localizeBody(entry, lang)`
  en `i18n.ts` (o en `util/`) las convierte en una línea, y el día que la regla
  cambie hay un solo sitio.
- **Esfuerzo**: S · **Riesgo**: bajo

```ts
// i18n.ts
export function localizeBody(body: string | string[], lang: Language) {
  const blocks = Array.isArray(body) ? body : [body];
  return blocks[languages.indexOf(lang)] || blocks[0] || '';
}
```

### 2.2 · Documentar por qué existe el loader propio (no borrarlo)

- **Qué**: `content.config.ts` lleva 180 líneas de loader propio rotulado "pay
  no attention to it". **Es necesario y no reemplazable**: para partir el body en
  un bloque por idioma (`body: content.split('---')`) hace falta llamar a
  `store.set({ body })`, y el único sitio donde se puede es dentro de un loader.
  El `glob()` oficial de Astro 5 sólo expone `pattern`, `base` y `generateId` —
  **no da ningún hook para transformar el body** (verificado en la 5.1.7
  instalada). Sin loader propio, no hay markdown multilingüe.
- **Dónde**: `src/content.config.ts:123-315`
- **Por qué**: el problema no es que exista, es que el comentario **esconde** la
  razón en vez de darla. "Pay no attention to it" convierte una decisión de
  diseño legítima en algo que el siguiente que pase (o un agente) borrará por
  parecer copia gratuita. Sustituir el comentario por el porqué, y subirlo como
  invariante al AGENTS.md del proyecto ("el markdown se parte por `---` en un
  bloque por idioma; eso obliga al loader propio, porque el `glob()` oficial no
  deja tocar el body").
- **Esfuerzo**: S · **Riesgo**: bajo

### 2.3 · Adelgazar el loader a lo que de verdad usa (opcional)

- **Qué**: de las 180 líneas, la parte imprescindible es leer el fichero, parsear
  el frontmatter y `store.set({ body: content.split('---') })`. El resto es
  maquinaria de watch incremental y digest copiada del glob oficial. Restos
  concretos: `posixifyPath` sólo lo usa `posixRelative` (inlinable),
  `generateDigest` se recibe en el destructuring pero su uso está comentado
  (`// generateDigest(...)`).
- **Dónde**: `src/content.config.ts`
- **Por qué**: menos superficie que mantener. Pero el watch incremental es lo que
  hace rápido el `astro dev`, así que **no** es un borrado gratis: sólo quitar lo
  muerto (`generateDigest` sin uso, inlinar `posixifyPath`) es seguro; recortar
  el watch es un cambio con riesgo que sólo vale si molesta.
- **Esfuerzo**: S (lo muerto) / M (el watch) · **Riesgo**: bajo / medio

## 3. Arquitectura y diseño

### 3.1 · Tipar el contenido en vez de `any`

- **Qué**: `CareerTimeline` (`career: any[]`, `(item: any)`) y `ContentCards`
  (`entries: any[]`, `(entry: any)`) reciben colecciones sin tipo. Astro genera
  los tipos de las colecciones (`CollectionEntry<'career'>`,
  `CollectionEntry<'projects'>`…).
- **Dónde**: `components/organisms/CareerTimeline.astro:6` ·
  `components/organisms/ContentCards.astro:6,14,25,49`
- **Por qué**: son 45 `any` en total; estos dos componentes son el grueso. Con
  el tipo de colección, un cambio de schema se propaga y el editor ayuda. El
  AGENTS.md del mono pide preguntar antes de usar `any`; aquí la alternativa
  existe y es gratis.
- **Esfuerzo**: M · **Riesgo**: bajo

### 3.2 · `Markdown.astro` no debería declarar la lista de lenguajes

- **Qué**: el componente fija a mano los 15 lenguajes que Shiki carga. Va de la
  mano de 4.1 (crear el highlighter una vez): al extraerlo a un módulo, la lista
  vive en un solo sitio con nombre.
- **Dónde**: `components/molecules/Markdown.astro:12-15`
- **Esfuerzo**: S (con 4.1) · **Riesgo**: bajo

## 4. Rendimiento (lo pide su propio AGENTS.md)

### 4.1 · Crear el resaltador de Shiki una sola vez

- **Qué**: `createHighlighter({ langs: [15 lenguajes] })` corre **en cada
  render** de `<Markdown>`. Con ~24 posts × 2 idiomas + proyectos y charlas, son
  decenas de inicializaciones completas de Shiki en cada build.
- **Dónde**: `components/molecules/Markdown.astro:12`
- **Por qué**: es la ineficiencia más clara, y el AGENTS.md pone el rendimiento
  como principio. Shiki recomienda una instancia reutilizada; se saca a un módulo
  con un singleton y `Markdown` lo pide.
- **Esfuerzo**: S · **Riesgo**: bajo

```ts
// util/highlighter.ts
import { createHighlighter } from 'shiki';

const LANGS = ['javascript', 'typescript', 'html', 'css', 'json', 'bash',
  'shell', 'c', 'cpp', 'csharp', 'python', 'rust', 'go', 'java', 'text'];

let instance: ReturnType<typeof createHighlighter> | null = null;
export function getHighlighter() {
  return (instance ??= createHighlighter({ themes: ['github-dark'], langs: LANGS }));
}
```

```astro
--- // Markdown.astro
import { getHighlighter } from '@/util/highlighter';
const highlighter = await getHighlighter();
---
```

(Mismo patrón que el `singleExecution` que ya usa mud; aquí basta un módulo con
estado, que en un build es efímero y no viola nada.)

### 5.1 · README al día con Bun

- **Qué**: el README dice `npm install` / `npm run dev`; el proyecto va con Bun
  (lockfile y CI).
- **Dónde**: `README.md`

- **Esfuerzo**: S · **Riesgo**: bajo

### 5.2 · La CI podría construir, no sólo typechequear

- **Qué**: `ci-amatiasq.com.yml` corre `astro check` pero no `astro build`. Un
  fallo que sólo aparezca en build (un import de asset roto, un problema del
  loader) pasa la CI.
- **Dónde**: `.github/workflows/ci-amatiasq.com.yml`
- **Por qué**: en un sitio estático el build **es** el test. Es un `astro build`
  extra en el job. (En mud añadimos el build a la CI justo por esto.)
- **Esfuerzo**: S · **Riesgo**: bajo

## Plan de ataque

Del más barato y seguro al más invasivo. Cada paso es verificable con
`astro check` + `astro build`.

**Fase 1 — Quick wins (una tarde)**

1. `0.1` `Image.astro` usa `useTranslations` → arregla las imágenes en `/es/`.
2. `1.1` borrar `internals/i18n.ts`, ya sin usuarios.
3. `1.2` los tres warnings de `astro check`.
4. `4.1` + `3.2` Shiki una sola vez, con la lista de lenguajes en su módulo.
5. `2.1` `localizeBody()` en las 6 páginas.
6. `5.1` README con Bun · `5.2` build en la CI.

**Fase 2 — Limpieza con más superficie**

7. `3.1` tipar `CareerTimeline` y `ContentCards` con `CollectionEntry`.
8. `1.3` colores de marca al frontmatter de career.

**Fase 3 — El loader**

9. `2.2` sustituir "pay no attention to it" por el porqué real (el split del body
   por idioma obliga al loader propio), y subirlo al AGENTS.md como invariante.
   `2.3` quitar sólo lo muerto (`generateDigest`, inlinar `posixifyPath`); no
   tocar el watch incremental.

**Lo que no haría**

- **Borrar el loader propio.** Es lo que hace posible el markdown multilingüe;
  el `glob()` oficial no puede partir el body por idioma. Confirmado con el
  autor y en el typedef de Astro 5.1.7.
- Añadir cualquier framework de cliente, o "modernizar" el i18n a una librería.
  El sistema funcional basado en URL es simple y correcto.
- Tocar el CSS de `Layout.astro`: es grande pero es la hoja de estilos del
  sitio, y está ordenada.
